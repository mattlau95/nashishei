# UX Audit Report — nàshìshéi (Who Is That?)

> **Date:** 2026-06-30 · **Reviewed:** `/` (login, live at `localhost:5173`), `/s/:token` (Viewer, source), `Home.tsx`, `FaceNameList.tsx`, `ImageDetector.tsx`, `QCOverlay.tsx` (source) · **Tools run:** Lighthouse, axe-core, pa11y, manual source + keyboard-path review

---

## Executive summary

The naming/editing flow itself is well built — confirm dialogs, visible loading states, keyboard-operable box editing, and AI suggestions that stay overrideable are all already in place. The headline problem is architectural: the ~190MB ONNX face-recognition models load unconditionally on **every** route, including the login screen and the no-account share-link viewer that never run on-device ML at all — this alone produces a 26-second LCP and a 217MB page weight on first visit. Combined with two accessibility blockers (unlabeled login inputs, a contrast token that's wrong even in its own documentation) and a data-loss gap in the naming flow, that's **4 blockers, 3 important, 3 polish**.

**Snapshot**

- Live page measured: `/` (login) — only public, ML-independent route in the app
- Core Web Vitals (login page): LCP **26.0s** · TBT **1,510ms** · CLS **0** *(2026 thresholds: LCP <2.5s, INP <200ms, CLS <0.1)*
- Total page weight: **217,628 KiB** (~212MB), driven almost entirely by ONNX model downloads the login page never uses
- Accessibility (axe-core): 1 violation type, 4 occurrences (color contrast). Pa11y (WCAG2AA ruleset): 5 errors — 2 unlabeled form inputs, 1 redundant label error, 2 contrast failures
- Checklist items reviewed: ~25 directly evidenced (live + source); roughly 40/61 full checklist estimated passing — see caveats per finding

> **Methodology note:** Lighthouse ran against the Vite **dev server** (no production server was available without provisioning Postgres/Docker for this pass), so JS-minification/bootup-time sub-metrics are dev-mode noise and are *not* reported as findings below. The page-weight/LCP finding is **not** a dev artifact — the ONNX model files are real network payload in any environment.

## What passed

- **The naming/editing flow's AI UX is genuinely strong** — detection results are fully editable (add/move/resize/delete boxes), name suggestions are explicitly framed as suggestions with confirm (✓) / dismiss (×) actions rather than auto-applied, and every async step shows a visible state ("Detecting faces…", "Saving…", "Generating link…", "✓ Copied").
- **Destructive actions are handled correctly**: the newly shipped photo-delete (MAT-530) has a confirm step, scopes by account server-side, and the button disables mid-request to prevent double-fire.
- **`QCOverlay`'s custom drag interaction has unusually good keyboard support** for a hand-built component — Tab-cycle between boxes, arrow-key nudge, Backspace/Delete to remove, Escape to deselect — well above the bar for this kind of UI.
- **The color token system is mostly correct and well-documented**: `docs/color-tokens.csv` lists verified AA/AAA contrast ratios for text, muted text, error red, and overlay labels — only `--color-blue` (not even in that file) is wrong.
- **Bulk-name parsing supports Chinese punctuation** (`，`、`) alongside Western commas — a thoughtful, audience-specific touch per the stated romanized-name persona.
- **Auth-expiry is handled consistently**: a global `session-expired` event (`frontend/src/lib/api.ts`) fires on any 401 from any call site, so there's one consistent path back to the login screen rather than scattered silent failures.
- **Tap targets are deliberately sized** via a shared `--tap-target` CSS variable used throughout, meeting WCAG 2.2's Target Size guidance.
- CLS measured **0** on the login page — no layout-shift problems observed.
